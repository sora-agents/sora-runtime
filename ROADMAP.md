# S-ORA Implementation Roadmap

Tracks progress from tooling setup through a working implementation of the design in [README.md](README.md) and [EXAMPLES.md](EXAMPLES.md). See [docs/adrs/](docs/adrs/) for why specific decisions were made.

Ordering principle for Phase 3: **fakes and determinism first, real networks and real models last.** Both LLM calls and real external adapters make tests slow, flaky, and non-reproducible ŌĆö deferring them keeps the test suite fast and deterministic for as long as possible. Phase 2 is the deliberate exception ŌĆö see its own note below.

## Phase 2 ŌĆö Walking skeleton against ARE

Goal: get one full ObserveŌåÆAct decision cycle running against a real external
adapter as fast as possible, to surface conceptual or design gaps in the
README/EXAMPLES.md sketch before investing in the full TDD build-out. This is
the one deliberate exception to "fakes and determinism first": ARE's MCP
server ([EXAMPLES.md](EXAMPLES.md#example-evaluating-a-s-ora-agent-on-are-meta))
runs a fixed scenario locally as a scripted, deterministic subprocess ŌĆö not a
flaky live network dependency ŌĆö so pulling it forward here doesn't cost the
reproducibility the TDD-first ordering is protecting. Code from this phase is
explicitly throwaway/minimal: it gets replaced and properly test-driven in
Phase 3, not polished here. Test coverage for this phase is a single
integration-level test, not unit tests per layer.

## Phase 4 ŌĆö Advanced Features

v0.1.0 tags mid-phase, not at a phase boundary: Track A and Track B's B1/B2 gate the release; WoT (Track C), B3 (relocated below it), and advanced reasoning (Track D) are post-tag ŌĆö see the marker after Track B.

### Track A ŌĆö Dynamic Environments (A1 gates A2; A3 sequenced after A1 for a real consumer; A4 independent/parallel)

- [ ] A1. Blocked-state machinery ŌĆö the `_suspend_`/`_resume_` internal actions and the signal-satisfies-wait judgment (Situate/Reflect) that transitions a `blocked` activity back to `ready`. Deferred from Phase 3 D5: no v0.1.0 scenario needs a manual that waits on a specific signal. **Owns signal lifecycle** (surfaced in D3): signal percepts are fire-and-forget and `_filter_` deliberately never drops them, so they accumulate in `wm.perceptions` until *consumption-driven* retention/eviction lands here ŌĆö a signal leaves working memory once the activity it concerns has matched/consumed it, or once it's provably orphaned (target activity terminated, or none was ever waiting). Needs a signalŌåÆactivity association (today `focused_tools` is agent-global, with no per-activity focus ownership), so it couples to how focus is scoped. **Also reconsider property/signal storage here (from D1a).** Today properties and signals share one flat `wm.perceptions: list[Percept]`, but they want opposite collection semantics ŌĆö properties are a replace-by-`(source, name)` snapshot (one entry, last value wins), signals are an append log that must keep duplicates (two `new_email` from the same source both survive). D1a's `DefaultObserveStrategy._snapshot_properties` bridges that gap with a side index (`(source, name) -> list position`) to get O(1) replace-by-key *over a list that also holds un-keyable signals*. That indirection is the smell: it must stay consistent with the list, it reads `p.payload.name` off a percept whose `payload` is typed `Any` (only correct by construction), and it exists purely because the two lifecycles are co-mingled in one list. The principled fix is to **split the storage** ŌĆö model properties as a keyed store (`dict[(source, name), ...]`) and signals as their own list ŌĆö which drops the index, the `Any` access, and the flat-list-two-semantics smell, and matches the state-vs-event-log distinction this item already draws. Deferred here (not into D1a's small Observe change) because it reshapes `WorkingMemory` and ripples to `_filter_`, `UnfocusAction`, tests, and README/EXAMPLES's single-percept-stream model ŌĆö adjacent to [ADR-0012](docs/adrs/0012-percepts-vs-messages.md), so likely ADR-worthy ŌĆö and this is the item already reworking how signals are stored and consumption-evicted.
- [ ] A2. Full ARE simulation-engine integration (beyond the static tool surface). Today S-ORA reaches ARE through the **MCP tool surface** ŌĆö the MCP server exposes a static snapshot of app *initial* state and never runs `Environment.run`, so the scenario **event timeline** (mid-run email injections, task delivery via `AgentUserInterface.send_message_to_agent`, follow-ups) never fires; that's why the gaia2 showcase must **seed** its inbox directly (`are_server.py`) rather than have ARE deliver it. To reproduce a real Gaia2-style *dynamic* scenario (and the signal-driven replanning against real ARE that the fakes-based `test_gaia2_reproduction.py` currently stands in for), run the ARE `Environment` (background event loop, advancing time) and **bridge its notification/AUI layer into S-ORA**: `NotificationSystem`/AUI `USER_MESSAGE` ŌåÆ `MessageTransport`, app-write `resource_updated` ŌåÆ `signal_sink`, plus the validator (oracle events) for scoring. Two shapes: (a) an in-process `WorkspaceAdapter` over live ARE apps (bypasses MCP), or (b) a custom launcher that `Environment.run`s in a thread and points `ARESimulationMCPServer` at the live apps (keeps the MCP adapter). Medium-large; the notification bridge is the crux (EXAMPLES.md flagged `USER_MESSAGE` routing as "not wired yet"). Couples to the blocked-state machinery (a `send_message_to_user` ask-user maps onto `_suspend_`/`_resume_`). **Gates v0.1.0 deliberately** despite being the largest single item here: it ships a second real, dynamic testing environment alongside WoT, and is the strongest concrete showcase for reaching out to the ARE/Meta team as v0.1.0's first external audience ŌĆö accepted schedule cost, no tag urgency otherwise.
- [ ] A3. `DecisionCycle.interrupt()` ŌĆö implement the 10ms hard-interrupt path (currently `NotImplementedError`). Decision-cycle preemption machinery, not CLI sugar: a `Signal` pushed to `signal_sink` can preempt the current phase, so `interrupt()` lives next to it. Its first real consumer is A1's blockedŌåÆready resume (an urgent signal should wake a waiting activity immediately, not at next tick) ŌĆö not the CLI: `TerminalSession` queues stdin as a `Message` (cooperative `communication.receive()` drain in Observe), not a `Signal`, so CLI input never drives this path.
- [ ] A4. Dynamic environments ŌĆö handle a workspace whose live tool set has drifted since it was last joined. Today `restore()` is a pure snapshot reconstruction (record-driven, skips `discover()`, no write-back): a newly-added tool is silently invisible, a removed tool becomes a stale handle whose failure is deferred (adapter-dependent), and a changed manual restores stale ŌĆö with **no reconciliation/refresh action and no drift detection**, so an agent that restores into a changed environment silently runs on a stale world model. Decide and build the fix (leading candidate: an explicit, agent-driven `refresh`/`resync` external action that re-`discover()`s a joined workspace, diffs against the registry + records, applies the register/deregister delta, and re-persists ŌĆö keeping `restore()` fast and pure). Pin the removed-tool `connect()` semantics (eager-validate vs. lazy-rebuild) in an adapter ADR; couples to the MCP-adapter hardening. Full analysis, drift table, and options in [docs/restore-drift-reconciliation.md](docs/restore-drift-reconciliation.md).

### Track B ŌĆö LLM Provisioning (B1 before B2 recommended ŌĆö avoid wiring per-strategy injection around a client shape that's about to be reshaped)

- [ ] B1. Two-axis LLM provider decoupling (Hermes-informed). E2 ships only the seam ŌĆö the wire-format-neutral `LLMClient` Protocol plus one concrete `AnthropicLLMClient`. **Motivated for v0.1.0, not speculative:** OpenAI and Gemini are concrete v0.1.0 provider targets (Gemini specifically for multimodal ŌĆö see B3), so three real wire formats justify splitting into a declarative provider descriptor ├Ś a behavioral transport over a normalized response with a `provider_data` escape hatch, instead of each provider re-deriving the seam ad hoc. Reason turning *conversational* later (the README-foreseen SituateŌåÆReasonŌåÆAct fusion into one multi-turn/tool-calling model call) is a further trigger, not the one motivating it now. **Must reuse `bootstrap`/`agent.yaml` + `import_object` for selection ŌĆö not a parallel `register_provider` registry** (that would fight the "all wiring centralized in bootstrap" invariant, ADR-0013). Likely ADR-worthy (a genuine live alternative exists now).
- [ ] B2. Surface the `LLMClient` to every phase strategy (per-strategy client injection). Today the model is private to `ProceduralMemory` (`self._llm`); the other phase strategies (Observe/Reflect/Situate/Act) have no handle to a configured model, so E2's model-backed reasoning reaches it only through `infer()`'s plan-shaped hole ([docs/reasoning-strategy-extension.md](docs/reasoning-strategy-extension.md), Gap 1). Fix: inject an `LLMClient` **per strategy** at construction ŌĆö a per-phase `model:` in `agent.yaml`, built in `bootstrap`, extending the `Strategies`-bundle and model-is-config patterns to *model-per-phase-is-config*. Multi-model then falls out for free (e.g. Opus for Reason, a cheaper model for Reflect summaries), and a developer's custom strategy is simply handed its own configured client at construction. Prefer this over a single ambient `cycle.llm`, which invites an ambient-mutable-client ambiguity (a shared client is at most a convenience fallback for the common one-model agent, not the primitive). `ProceduralMemory` keeps owning its (Reason-side) client; instances are shared per [ADR-0013](docs/adrs/0013-shared-instances-narrow-dependencies.md). This does **not** make `infer()` redundant ŌĆö it stays procedural memory's *packaged* planning query (prompt build + parse/validate into `Plan`/`Step` + reuse across runs), distinct from the raw reasoning a phase does with its own client (interpret a percept, summarize an episode, derive a goal). The risk to guard with guidance, not structure, is *bypass temptation*: hand-rolling planning against the raw client and losing the parse/reuse.

- [ ] C2. Extend `Manual` reconciliation to WoT Thing Descriptions ŌĆö the WoT half split out of the MCP-scoped reconciliation. - [ ] C3. Cross-workspace tool sharing ŌĆö decide whether the same tool may be a member of two workspaces. Today `_register` rejects any duplicate `Tool.id` ([ADR-0014](docs/adrs/0014-tool-identity-globally-unique.md)), so a tool cannot appear in two workspaces. That's correct for the A&A/CArtAgO *containment* view (exclusive membership, one owner) but too strict for the Web/hypermedia *index* view, where a workspace is a logical container and two workspaces ŌĆö each on its own server ŌĆö could legitimately reference a tool running on a third (via `Tool.address` override). ADR-0014 already put identity on a global URI, which surfaces a tension: it wants global identity *and* rejects duplicate ids as collisions. Leading candidate: split by connection ownership ŌĆö connection-owned tools (`address is None`) stay exclusive; self-addressed tools (own global URI) may be referenced from multiple workspaces, with `_register` admitting a duplicate id only on canonicalized-equal address, `_workspace_tools` many-to-many, and refcounted deregistration. Couples to the `Manual` reconciliation (which manual wins for a shared tool) and the dynamic-environments refcount bookkeeping. Not needed for v0.1.0 (ARE has no shared tools); realize with the WoT adapter + two-agent lab, as an ADR refining ADR-0014. Full analysis and options in [docs/cross-workspace-tool-sharing.md](docs/cross-workspace-tool-sharing.md).
- [ ] C4. Multimodal model support ŌĆö especially for perception (Observe). `LLMClient.complete(system, prompt) -> str` is text-only and cannot carry an image, so the README-foreseen **LLM-backed Observe** that interprets raw perception itself (*"describing a camera snapshot"*) is not yet expressible. Fix: a **richer `LLMClient` Protocol** carrying multimodal (and tool-calling) content ŌĆö the same normalized-client seam as B1's two-axis decoupling. Modality is a *Protocol-shape* problem, **orthogonal** to *which model per phase* (B2's per-strategy injection): per-phase wiring picks *which* model, the richer Protocol handles *what modality* ŌĆö so don't try to solve multimodal by adding more clients. Depends on B1 + B2 above. **Real trigger now:** C1's reworked `video-stream` device (raw camera feed, no on-device summarization) is what finally gives Observe a percept that actually needs vision ŌĆö v0.1.0's ARE email/calendar never did, and the WoT scenario's old toy description didn't either.
